2. Grundlegende Besonderheiten von Linux

🔵 Alles ist eine Datei

Geräte (/dev/sda), Prozesse (/proc), Netzwerk (/sys) – alles wird als Datei behandelt. Einheitliches Interface!

🔵 Multiuser-System

Mehrere Benutzer gleichzeitig aktiv. Jeder hat eigene Rechte. Root = Superuser mit vollen Rechten.

🔵 Groß-/Kleinschreibung

Datei.txt ≠ datei.txt. Linux unterscheidet strikt zwischen Groß- und Kleinbuchstaben!

🔵 Distributionen

Ubuntu, Debian, Fedora, Raspbian (Raspberry Pi OS) – alle basieren auf dem Linux-Kernel, unterscheiden sich in Paketmanager & GUI.

🔵 Kein Laufwerksbuchstabe

Alles startet bei / (Root-Verzeichnis). USB-Sticks werden gemountet z. B. unter /mnt/usb.

🔵 Shell / Terminal

Bash ist die Standard-Shell. Befehle werden als Text eingegeben. Sehr mächtig für Automatisierung & Scripts.

sudo usermod -aG gpio pi  # pi zur gpio-Gruppe hinzufügen

🧠 Quiz – Linux (zum Anklicken)

❓ Was bedeutet chmod 644 datei.txt?
Besitzer: rw- (6=4+2), Gruppe: r-- (4), Andere: r-- (4). Der Besitzer kann lesen und schreiben, alle anderen nur lesen.
❓ Was ist der Unterschied zwischen /bin und /sbin?
/bin enthält Programme für alle Benutzer (z. B. ls, cp). /sbin enthält System-Programme, die nur root ausführen darf (z. B. ifconfig, reboot).
❓ Ich bin in /home/pi/projekt. Wie komme ich mit einem relativen Pfad nach /home/pi/scripts?
cd ../scripts – mit .. eine Ebene rauf nach /home/pi, dann in scripts.
🍓 Raspberry Pi

1. Belegungs- und Schaltplan

📋 Belegungsplan (Pinout) Zeigt welcher GPIO-Pin welche Funktion hat. Der Raspberry Pi hat 40 Pins (GPIO, 3.3V, 5V, GND, SPI, I2C, UART).
🔌 Schaltplan (Schematic) Zeigt wie externe Bauteile (LEDs, Sensoren, Widerstände) mit den GPIO-Pins verbunden werden. Enthält: Widerstände, Spannungsversorgung, Signalleitungen.
⚡ GPIO-Spannungspegel: 3.3V ! Die GPIO-Pins des Raspberry Pi arbeiten mit 3,3V – NICHT 5V! Ein 5V-Signal direkt am GPIO-Pin kann den Pi zerstören!

Typischer GPIO-Aufbau (LED):

  GPIO (3.3V) ──┬── [R = 330Ω] ──┬── LED(+) ──┬── GND
                │                │            │
                └── High = LED an└── Low  = LED aus

Schaltplan-Symbole:
  ──[  ]──  Widerstand
  ──|>|──   LED (Pfeil = Stromrichtung)
  ──┴──     GND (Masse)
  VCC       Versorgungsspannung

3. Spannung an GPIO-Eingängen berechnen

⚡ Spannungsteiler-Formel
Wenn zwei Widerstände R1 und R2 in Reihe an VCC (3.3V) liegen:

U_GPIO = VCC × R2 / (R1 + R2)

U_GPIO = Spannung am GPIO-Pin (= an R2)
Beispiel:
VCC = 3.3V, R1 = 10kΩ, R2 = 10kΩ
U_GPIO = 3.3V × 10k / (10k + 10k)
U_GPIO = 3.3V × 0.5 = 1.65V

Direktanschluss (kein Teiler):

GPIO direkt an 3.3V → HIGH (logisch 1)
GPIO direkt an GND → LOW (logisch 0)

Schwellwert Raspberry Pi:
HIGH: > 1.8V | LOW: < 0.8V

4. Stromstärke berechnen (Ohmsches Gesetz)

🔢 Ohmsches Gesetz

U = R × I
I = U / R
R = U / I

U = Spannung [V] | R = Widerstand [Ω] | I = Stromstärke [A]
Beispiel: LED-Widerstand berechnen
GPIO = 3.3V, LED-Spannung U_LED = 2V, gewünschter Strom I = 10mA = 0.01A

Spannung am Widerstand: U_R = 3.3V − 2V = 1.3V
R = U_R / I = 1.3V / 0.01A = 130Ω
→ Nächster Standardwert: 150Ω oder 220Ω

5. Widerstände prüfen & dimensionieren

⚠️ GPIO-Strom-Limit des Raspberry Pi Max. 16 mA pro GPIO-Pin, max. 50 mA gesamt über alle GPIOs!
Formel für Mindestwiderstand: R_min = (VCC − U_Bauteil) / I_max
SituationBerechnungErgebnis
LED (U=2V, I_max=20mA)R = (3.3−2) / 0.0265Ω → min. 68Ω nehmen
LED (I=10mA, sicher)R = (3.3−2) / 0.01130Ω → 150Ω nehmen
Pegel-Schutz (kein Bauteil)R = 3.3 / 0.016ca. 200Ω → Schutzwiderstand

6. Pull-up und Pull-down Widerstände

⬆️ Pull-up Widerstand
GPIO wird über R (z.B. 10kΩ) mit VCC (3.3V) verbunden.

→ Ohne Tasterdruck: GPIO liest HIGH (3.3V)
→ Mit Tasterdruck (GND): GPIO liest LOW (0V)

  3.3V ──[10kΩ]──┬── GPIO
                 │
              [Taster]
                 │
               GND
Logik: Normal HIGH, bei Druck LOW
⬇️ Pull-down Widerstand
GPIO wird über R (z.B. 10kΩ) mit GND verbunden.

→ Ohne Tasterdruck: GPIO liest LOW (0V)
→ Mit Tasterdruck (3.3V): GPIO liest HIGH

  3.3V
    │
 [Taster]
    │
    ├── GPIO
    │
  [10kΩ]
    │
   GND
Logik: Normal LOW, bei Druck HIGH
🧠 Merkhilfe Pull-up: zieht den Pin nach oben (HIGH) → Taster drückt ihn runter (LOW)
Pull-down: zieht den Pin nach unten (LOW) → Taster hebt ihn hoch (HIGH)
Ohne Pull-Widerstand: „floating" = undefinierter Zustand → fehlerhafte Erkennung!

🧠 Quiz – Raspberry Pi (anklicken)

❓ Du hast eine LED mit U_LED = 2.2V. GPIO = 3.3V, Strom = 10mA. Wie groß muss der Vorwiderstand mindestens sein?
U_R = 3.3 − 2.2 = 1.1V
R = U_R / I = 1.1V / 0.01A = 110Ω → Standardwert: 120Ω oder 150Ω nehmen.
❓ SSH auf den RASPI (192.168.2.59) klappt nicht. Ping klappt. Welche OSI-Schicht hat das Problem und was prüfst du?
Ping = Layer 3 (IP) funktioniert. Problem ist also Layer 4+ (Port 22 zu?) oder Layer 7 (SSH-Dienst läuft nicht?).
Prüfen: systemctl status ssh auf dem RASPI, netstat -tuln | grep 22.
❓ Wozu braucht man Pull-Widerstände? Was passiert ohne sie?
Ohne Pull-Widerstand ist ein GPIO-Eingang „floating" – er hat keinen definierten Pegel. Kleine Störungen (elektromagnetische Einstreuung) lassen den Pin zufällig HIGH oder LOW werden → unzuverlässige Erkennung des Tastersignals.

3. Cloud – Fog – Edge Computing

☁️ Cloud Computing

Verarbeitung in entfernten Rechenzentren.

  • Große Rechenleistung
  • Hohe Latenz (ms bis Sekunden)
  • Für Langzeitanalyse, KI-Training

Beispiel: Azure IoT Hub analysiert Produktionsdaten

🌫️ Fog Computing

Verarbeitung in lokalen Servern (nahe am Shopfloor).

  • Mittlere Latenz (ms-Bereich)
  • Lokales Netzwerk
  • Für Datenaggregation, lokale Auswertung

Beispiel: OPC UA Server im Technikraum

⚡ Edge Computing

Verarbeitung direkt am Gerät / an der Maschine.

  • Sehr niedrige Latenz (<1ms)
  • Kein Netzwerk nötig
  • Für Echtzeit-Regelung, Anomalieerkennung

Beispiel: Raspberry Pi am Sensor = Edge Device

🧠 Merkhilfe: Wo wird verarbeitet? Edge = am Gerät selbst (schnellste) → Fog = lokal im Netz (mittel) → Cloud = weit weg im Internet (langsam aber mächtig)

6. Von der Pyramide zur vermaschten Struktur

🏔️ Klassische Pyramide

  • Strenge Hierarchie von oben nach unten
  • Jede Ebene kommuniziert nur mit der Nachbarebene
  • Langsame Kommunikation durch viele Ebenen
  • Schwer skalierbar, Silos

🕸️ Vermaschte Struktur (IIoT)

  • Jede Ebene kann direkt mit jeder kommunizieren
  • Sensor sendet direkt in die Cloud (Edge → Cloud)
  • Schnellere Reaktionen, mehr Flexibilität
  • Grundlage für Industrie 4.0 / IIoT
Vorteile der vermaschten Struktur: Direkte Kommunikation (Sensor → Cloud) | Schnellere Auswertung | Keine „Flaschenhälse" durch Mittelebenen | Bessere Skalierbarkeit | Grundlage für KI-basierte Produktion (Predictive Maintenance)

7. SCADA-Systeme

SCADA = Supervisory Control and Data Acquisition
System zur Überwachung, Steuerung und Datenerfassung von industriellen Prozessen und Anlagen über große geografische Bereiche.

Aufgaben von SCADA:

  • Echtzeit-Visualisierung von Prozessen (Dashboards)
  • Alarmierung bei Störungen (E-Mail, SMS)
  • Historische Daten aufzeichnen & archivieren
  • Fernbedienung von Anlagen (> Fernsteuerung)
  • Berichterstellung für Management

Eigenschaften:

MerkmalAusprägung
ReaktionszeitSekunden bis Minuten (nicht Echtzeit)
BereichGanze Anlage, Fabrik, Region
ProtokolleOPC UA, Modbus, DNP3
BeispieleSiemens WinCC, Wonderware, Ignition
⚠️ SCADA ≠ Echtzeit-Steuerung! SCADA überwacht und visualisiert – die eigentliche Echtzeit-Steuerung übernimmt die SPS (Ebene 2). SCADA sitzt auf Ebene 3.

🧠 Quiz – Vertikale Kommunikation

❓ Was ist der Unterschied zwischen Edge und Fog Computing?
Edge: Verarbeitung direkt am Gerät/Sensor – keine Netzwerklatenz, Echtzeit möglich. Fog: Verarbeitung auf einem lokalen Server/Gateway im Netz – etwas mehr Latenz, aber mehr Rechenleistung und Datenaggregation über mehrere Geräte möglich.
❓ In welcher Ebene der Automatisierungspyramide sitzt der OPC UA Server aus dem Netzwerkdiagramm?
Der OPC UA Server sitzt auf Ebene 3 (SCADA/Prozessleitebene). Er sammelt Daten von der SPS (Ebene 2) und stellt sie der Office-IT (Ebene 4/5) zur Verfügung – er ist das Bindeglied zwischen Shopfloor und Office.
❓ Warum ist die vermaschte Struktur besser als die klassische Pyramide?
In der Pyramide muss Kommunikation durch alle Ebenen – das dauert und erzeugt Engpässe. In der vermaschten Struktur kann ein Sensor direkt in die Cloud senden (Edge → Cloud), was schnellere Reaktionen und flexiblere Architektur ermöglicht. Besonders wichtig für IIoT und Industrie 4.0.
